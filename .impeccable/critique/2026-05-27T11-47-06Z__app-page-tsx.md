# Anti-Patterns Verdict

**LLM assessment**: This is not a disaster. It is coherent, legible on desktop, and clearly about a benchmark workbench. But it does look like an AI-polished devtool landing page: dark navy surface, teal gradient headline, ambient glow orbs, repeated icon cards, and a dense faux dashboard mockup. The issue is not craftlessness; it is insufficient Keystone-ness.

The current page also conflicts with the loaded product context. Keystone is supposed to feel monolithic, precise, resilient, light-first with deliberate dark technical bands, using the hidden-K/KS glyph language and restrained blueprint blue. The live page is dark-first, teal/violet, and generic observability-coded. The current logo mark in nav is a rounded gradient square, not the structural Keystone glyph.

# Overall Impression

The page has a strong product narrative spine: hero, live-looking workbench, capability cards, comparison, trust, download. The biggest opportunity is to turn it from a polished placeholder into a credible cryptography infrastructure surface. For production in one pass, fix mobile layout, remove false actions, and make the evidence trail honest. The later installer/repo-coordination pass can then add actual binaries and release plumbing without the page feeling like a promise it cannot keep.

# What's Working

1. The app mockup gives the page an immediate product object. For this audience, visible benchmark data is more persuasive than abstract copy.
2. The information architecture is sensible on desktop: features, benchmarks, security, download, docs. The story sequence is easy to understand.
3. The copy mostly avoids quantum mysticism. It stays closer to benchmarking, reproducibility, and local execution, which is the right strategic territory.

# Priority Issues

**[P0] Mobile layout is visibly broken**

**Why it matters**: At a 390px viewport, the hero content and mockup render at roughly 857px wide. The headline, paragraph, second CTA, proof pills, and mockup are clipped to the right. A mobile visitor sees a broken production page before they can evaluate the product.

**Fix**: Make the hero grid track and children shrinkable: use `minmax(0, 1fr)` behavior, add `min-w-0` to the hero text and mockup wrappers, constrain the mockup with `max-w-full`, and stack CTAs/proof pills deliberately on small screens. Do not solve this with section-level clipping; the content itself needs a mobile composition.

**Suggested command**: `impeccable adapt`

**[P1] Production trust is undermined by live-looking dead actions**

**Why it matters**: The enabled Windows installer points to `#`; checksums point to `#`; GitHub/star/social/community links point to `#`. For security engineers, a dead checksum or fake download is not cosmetic. It damages credibility.

**Fix**: For this pass, make all unavailable installers explicitly unavailable or replace download CTAs with a truthful interim action such as release notifications, source preview, or benchmark report. Hide checksums until a real artifact exists. Route GitHub and community links only if the real destinations are available.

**Suggested command**: `impeccable harden`

**[P1] The brand misses the Keystone board**

**Why it matters**: The page currently reads dark observability SaaS. The supplied direction points toward light-first identity fields, dark technical panels, precise rules, restrained blue, and a monolithic K/KS glyph system. The current teal gradient square and gradient headline do not carry that identity.

**Fix**: Move the first viewport toward the board: light identity field, dark mockup/spec band, real Keystone glyph as structural layout device, solid blueprint-blue emphasis, and fewer teal/violet accents. Replace gradient text with a solid color and typographic weight/scale.

**Suggested command**: `impeccable polish`

**[P1] Evidence exists visually, but not traceably**

**Why it matters**: The audience arrives skeptical. The mockup metrics, security claims, "peer-reviewed code", "constant-time primitives", and checksums need provenance or careful framing. Right now the page looks certain, but the proof path is thin.

**Fix**: Add a compact evidence strip near the mockup or comparison: sample run label, scheme set, platform, commit/source placeholder if available, report link, and release verification policy. If data is mock/sample data, say so cleanly.

**Suggested command**: `impeccable clarify`

**[P2] Repeated card grids flatten the story**

**Why it matters**: Features has five equal cards; trust has five equal cards; comparison uses four equal chart cards; footer adds four link columns. The visual rhythm becomes "another grid" instead of a product argument.

**Fix**: Replace at least one grid with a more Keystone-specific structure: a verification pipeline, a benchmark specimen, a source-to-artifact chain, or a glyph-derived diagram. Keep groups to 3-4 visible items where possible.

**Suggested command**: `impeccable layout`

# Persona Red Flags

**Jordan (First-Timer)**: Jordan understands "Download Keystone" but then gets an enabled `.exe` link that goes nowhere and a checksums link that goes nowhere. "PQC", "ML-KEM", "ML-DSA", and "SPHINCS+" appear without lightweight grounding. Jordan can admire the page, but cannot confidently answer "what should I do next?"

**Casey (Distracted Mobile User)**: Casey sees clipped text and clipped buttons immediately. The desktop nav is hidden, but no mobile menu replaces it. The second CTA and proof pills extend offscreen, and the mockup is too wide for the viewport. This is the release-blocking persona failure.

**Priya (Cryptography Evaluator)**: Priya wants source, reproducibility, artifacts, and exact provenance. She sees strong claims and convincing charts, but not enough traceability. The dead checksum link is especially damaging because it signals verification while failing to provide it.

# Minor Observations

- Sticky nav overlaps anchor destinations. `#download` lands with the heading hidden behind the header; add scroll margin to anchored sections.
- Disabled installer buttons still look too active because they retain large teal fills. The visual state should be unmistakably unavailable.
- The newsletter form prevents default submit and gives no success, error, loading, or privacy expectation.
- Typography is serviceable, but IBM Plex is not aligned with the supplied Sohne-like brand board and is also a common technical default.
- The nav/logo mark should use the actual Keystone glyph as soon as possible. The current rounded gradient square is too generic.

# Questions to Consider

1. Until real installers exist, should the primary CTA be "Join release list", "View benchmark report", or "Follow source release"?
2. Should this one-pass production push include the light-first brand correction, or should it only fix mobile, dead links, and evidence honesty?
3. What proof can ship now: a real benchmark report, a Keystone proper repo link, a sample-run disclaimer, or a release verification policy?
